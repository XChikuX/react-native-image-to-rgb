# CLAUDE.md

Guidance for Claude (and other LLM coding agents) working in this repo.

## What this package is

`react-native-ai-image` is a React Native library that decodes an image
(`file://`, `content://`, http(s), `data:` URI, plain path) and packs the
pixels into a zero‑copy `ArrayBuffer` shaped for ML inference (TFLite, ONNX,
darknet/YOLO, etc.). The native fast path is built on
[Nitro Modules](https://nitro.margelo.com) — a single Swift class on iOS and a
single Kotlin class on Android, both implementing a generated `HybridObject`
spec.

This repo was forked from `react-native-image-to-rgb`. Every native identifier
has been deliberately renamed away from the legacy `ImageToRgb` / `imagetorgb`
namespace so both packages can be installed in the same app without colliding.
**Preserve that disambiguation in every change.**

## Source‑of‑truth files (edit these; nitrogen regenerates the rest)

| Layer | File | Notes |
|---|---|---|
| Nitro config | `nitro.json` | Drives nitrogen codegen. |
| Spec | `src/specs/AiImage.nitro.ts` | The HybridObject interface + DTOs. |
| JS wrapper | `src/index.ts` | Public API — keep stable, this is the npm surface. |
| iOS impl | `ios/HybridAiImage.swift` | Final Swift class, extends generated `HybridAiImageSpec`. |
| Android impl | `android/src/main/java/com/margelo/nitro/aiimage/HybridAiImage.kt` | Final Kotlin class, extends generated `HybridAiImageSpec`. |
| Android RN package | `android/src/main/java/com/aiimage/AiImagePackage.kt` | Triggers `NitroAiImageOnLoad.initializeNative()` at app start. |
| iOS pod | `NitroAiImage.podspec` | Pod name **must** equal `ios.iosModuleName` in `nitro.json`. |
| Android build | `android/build.gradle`, `android/CMakeLists.txt`, `android/gradle.properties` | namespace `com.aiimage`, lib `NitroAiImage`, gradle props prefixed `AiImage_`. |
| JNI shim | `android/src/main/cpp/cpp-adapter.cpp` | Calls `margelo::nitro::aiimage::initialize(vm)`. |
| Autolinking glue | `react-native.config.js` | Tells RN autolinker to instantiate `AiImagePackage`. |
| Post-nitrogen fixup | `scripts/strip-swift-header-hack.js` | Strips nitrogen's broken `SWIFT_INSTALL_OBJC_HEADER => "NO"` line. |
| Tests | `src/__tests__/index.test.ts` | Pure JS — mocks Nitro. Native code isn't exercised. |

`nitrogen/generated/**` is fully regenerated by `bun run specs`. **Do not
hand‑edit anything there** except the one documented workaround below.

## Naming rules (do not break)

To stay side‑by‑side installable with `react-native-image-to-rgb`, every
globally‑namespaced symbol uses the `AiImage` / `aiimage` family. Keep them
all in sync. If you change one, change all of them, **and** delete
`nitrogen/generated/` and re‑run `bun run specs`.

| Layer | Value |
|---|---|
| Nitro registry key | `"AiImage"` (in `NitroModules.createHybridObject(...)`) |
| Spec interface | `AiImage` (in `src/specs/AiImage.nitro.ts`) |
| iOS pod / Swift module | `NitroAiImage` |
| Android native lib | `libNitroAiImage.so` (`NitroAiImage` in CMake) |
| Android Gradle namespace | `com.aiimage` (-> `com.aiimage.BuildConfig`) |
| Android RN package class | `com.aiimage.AiImagePackage` |
| Android Kotlin hybrid pkg | `com.margelo.nitro.aiimage.HybridAiImage` |
| C++ namespace | `margelo::nitro::aiimage` |
| Gradle ext property prefix | `AiImage_` (in `android/gradle.properties`) |

`react-native-image-to-rgb` uses the matching `ImageToRgb` / `imagetorgb` /
`com.imagetorgb` set; never reintroduce any of those names here.

## Public JS API — stable surface

`src/index.ts` exports — these are what consumers depend on, do not rename
without a major version bump:

- `convertImage(uri, options?) -> Promise<ConvertResult>`
- `convertImageSync(uri, options?) -> ConvertResult`
- `toYoloInput(uri, size?) -> Promise<ConvertResult>` (darknet/YOLO preset)
- `toTFLiteInput(uri, opts?) -> Promise<ConvertResult>` (TFLite mobile preset)
- `convertToRGB(uri) -> Promise<number[]>` (back‑compat shim for the v0.1.x
  `react-native-image-to-rgb` API)
- `asTypedArray(result)`, `bytesPerSample(dt)`, `channelsFor(fmt)` helpers
- Re‑exported types: `AiImage`, `ConvertOptions`, `ConvertResult`,
  `PixelFormat`, `DataType`, `ChannelLayout`, `ResizeMode`, `CropRect`,
  `NormalizationPreset`, `Normalization`, and the deprecated `ImageToRgb`
  alias.

## Common workflows

```bash
# Regenerate Nitro spec output (after any change to src/specs/*.nitro.ts or nitro.json)
bun run specs                  # -> writes nitrogen/generated/**

# After regenerating, REAPPLY the iOS workaround in
# nitrogen/generated/ios/NitroAiImage+autolinking.rb (see "Known footguns").

# Type‑check + tests
node node_modules/typescript/bin/tsc --noEmit
node node_modules/jest/bin/jest.js

# Build the publishable JS bundle
bun run prepare                # runs nitrogen + react-native-builder-bob
```

The example app (`example/`) is currently on RN 0.73.3 and doesn't list
`react-native-nitro-modules` — it cannot exercise this library as‑is. Treat it
as legacy until it's bumped to match the library's RN 0.76.x target.

## Known footguns

1. **`SWIFT_INSTALL_OBJC_HEADER = NO` in the generated iOS autolinking rb.**
   Nitrogen 0.35.7 unconditionally emits this in
   `nitrogen/generated/ios/NitroAiImage+autolinking.rb`. It disables the
   `NitroAiImage-Swift.h` header — which the generated
   `NitroAiImage-Swift-Cxx-Umbrella.hpp` `#include`s — and causes the iOS
   build to fail with *"NitroAiImage's autogenerated Swift header cannot be
   found"*, cascading into a wall of
   `HybridAiImageSpec_cxx` / `Func_void_ConvertResult` / `NitroAiImageAutolinking`
   "incomplete type" / "not a class" errors.

   **This is handled automatically.** Both `bun run specs` and `bun run prepare`
   run `scripts/strip-swift-header-hack.js` immediately after nitrogen
   regenerates the file. You do not need to hand-edit anything.

2. **Pod name == Swift module name == `ios.iosModuleName`.** The C++ side
   imports `NitroAiImage-Swift.h` and references the `NitroAiImage::`
   Swift‑module namespace; if the podspec is named anything else (e.g.
   `react-native-ai-image`), the Swift module ends up named
   `react_native_ai_image` and *every* generated bridge type fails to
   resolve. Always keep `NitroAiImage.podspec`'s `s.name = "NitroAiImage"`.

3. **Don't re‑register `"ImageToRgb"`.** The Nitro HybridObject registry is a
   process‑global map keyed by string. If you rename the spec interface back
   to `ImageToRgb` and a consumer also has `react-native-image-to-rgb`
   installed, the second package to register throws at startup.

4. **Android namespace drives `BuildConfig`.** `namespace "com.aiimage"` in
   `android/build.gradle` is what produces `com.aiimage.BuildConfig` and
   keeps Gradle's D8 dex merger from colliding with
   `com.imagetorgb.BuildConfig` from the legacy package. Reverting it brings
   back the duplicate‑class error from the user's bug report.

5. **`react-native-builder-bob` ships `lib/` to npm.** If you add new public
   exports in `src/`, run `bun run prepare` before publishing or consumers
   will see `undefined` imports.

## When in doubt

- Match the surrounding code style (Prettier config in `package.json` — 2 spaces, single quotes, no semicolons, trailing commas `es5`).
- Don't introduce new dependencies; this library is intentionally tiny.
- Don't touch `example/ios/Pods/`, `example/ios/Podfile.lock`, or anything under `nitrogen/generated/` *other than* the documented iOS workaround.
- Native behavior changes need to be implemented twice — once in Swift and once in Kotlin — and the spec in `src/specs/AiImage.nitro.ts` is the contract for both.
